Resumen General de la Versión

Esta actualización mejora la forma en que el sistema procesa los pedidos que contienen varios productos iguales (como hamburguesas o combos repetidos). Ahora el sistema puede identificar correctamente qué condimentos o cambios adicionales pertenecen a cada producto individual. Esto evita confusiones en la preparación del pedido y asegura que el cliente reciba exactamente lo que solicitó.

Análisis Técnico Detallado

api/src/platforms/interfaces/pedidosYa.js

Análisis General

Se ha corregido una limitación en la lógica de mapeo que utilizaba find() sobre la colección de subitems basándose solo en el SKU. Esto provocaba que, en pedidos con múltiples instancias del mismo SKU, todas las personalizaciones (extras y remociones) se asignaran incorrectamente a la primera instancia encontrada.

Cambios Detallados

Gestión de Colecciones y Asignación por Estado de Personalización

Explicación: La implementación ahora utiliza filter() para obtener todas las instancias de un SKU. Se introdujo una búsqueda secundaria de emptyItem que prioriza artículos sin personalizaciones previas. Además, se añadió parseFloat para normalizar el precio de los extras, evitando errores de tipo si el valor de entrada es un string.

Código Después: 

if (extraRemove.remoteCode.slice(-1) === 'E') {
  const targetSku = extraRemove.remoteCode.split('+')[1].slice(0, -1);
  const found = subitems.filter(r => r.sku === targetSku); // Obtiene todas las instancias del SKU
  const extraObj = {
    itemType: 'COMPONENT',
    itemId: extraRemove.id,
    sku: extraRemove.remoteCode.split('+')[0],
    itemDescription: extraRemove.name,
    price: parseFloat(extraRemove.price) || 0, // Asegura precio numérico
    discount: 0,
    quantity: detail.quantity
  };

  if (found.length > 1) {
    // Busca específicamente una instancia que aún no tenga extras asignados
    const emptyExtrasItem = found.find(r => r.customizations.extras.length === 0);
    if(!emptyExtrasItem)
      subitems.find(r => r.sku === targetSku).customizations.extras.push(extraObj)
    else     
      emptyExtrasItem.customizations.extras.push(extraObj);
  }
  else
    subitems.find(r => r.sku === targetSku).customizations.extras.push(extraObj)
}
// Aplicada lógica equivalente para componentes removidos (Caso "O")

Conclusión del análisis

El cambio de una búsqueda escalar a una gestión de conjuntos permite procesar correctamente estructuras de pedidos anidadas y redundantes, eliminando errores de solapamiento de datos en el integrador de PedidosYa.
